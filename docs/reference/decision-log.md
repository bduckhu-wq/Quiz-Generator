## ADR-001: 采用 Skill + Workflow 融合架构

**日期**：2026-04-28  
**状态**：✅ 已采纳  
**决策者**：技术团队

### 背景

需要在"策略灵活性"和"执行可控性"之间找到平衡。

### 决策

采用 **OpenClaw Skill（策略层）+ LangGraph Workflow（执行层）** 融合架构。

**架构图**：
```
教研人员修改 Skill（改规则） → Skills/exam_skill/config.yaml
                ↓
开发人员维护 Workflow（改流程） → Workflows/exam_workflow/graph.py
```

### 理由

| 维度 | 纯 Skill | 纯 Workflow | 融合方案 |
|-----|----------|-------------|---------|
| 灵活性 | ✅ 高 | ⚠️ 低 | ✅ 策略灵活 |
| 可控性 | ⚠️ 低 | ✅ 高 | ✅ 流程确定 |
| 可维护性 | ✅ 易改规则 | ⚠️ 需改代码 | ✅ 分离 |

### 后果

- 教研人员可独立迭代策略（修改 YAML 配置）
- 开发人员维护稳定的执行流程（LangGraph）
- 支持 A/B 测试不同版本的 Skill

## ADR-002: 题库优先，AI 补充策略

**日期**：2026-04-28  
**状态**：✅ 已采纳

### 背景

AI 生成题目质量不稳定，真题库质量有保障。

### 决策

**优先使用题库（80-90%），AI 仅补充缺口（10-20%）**。

### 流程

```
参数确认 → 题库检索 → 分析缺口 → AI 补充 → 合并组卷
```

### 理由

- 真题质量高，教师信任度高
- AI 仅用于补充，降低风险
- 明确标注来源（题库 vs AI 生成）

## ADR-003: 会话管理使用 JSON 文件

**日期**：2026-04-27  
**状态**：✅ 已采纳  
**决策者**：后端团队

### 背景

MVP 阶段需快速验证产品逻辑，避免引入数据库依赖。

### 决策

使用 JSON 文件存储会话（`sessions/{session_id}.json`）。

### 理由

- 无需引入 Redis/PostgreSQL
- 会话数据量小（1-5KB），文件读写足够
- 可平滑迁移至数据库

### 迁移计划

若满足以下条件之一，迁移至 Redis：
- 单用户会话数 > 100
- 并发用户 > 10

## ADR-004: 参数面板采用统一展示

**日期**：2026-04-27  
**状态**：✅ 已采纳

### 背景

原设计区分"已确认"和"AI 默认"参数。

### 决策

**所有参数统一展示，不区分视觉样式**。

### 理由

- 避免认知负担（无需区分参数来源）
- 教师可自由修改任何参数
- 简化前端逻辑
